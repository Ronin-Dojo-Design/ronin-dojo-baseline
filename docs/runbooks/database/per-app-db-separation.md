# Per-App Database Separation Runbook

The operational rules for [ADR 0038](../../architecture/decisions/0038-per-product-database-separation.md)
— **one database per product**. Each app in this monorepo owns its own `prisma/` schema, its own
`DATABASE_URL`, and its own migrations. There are **no cross-product foreign keys** and **no shared
Prisma package**. Components stay shared (`packages/ui-kit`); *data* does not.

## The rule (the guardrail)

> **A new product gets its own database.** If you are standing up a new app under `clients/*` (or a new
> top-level product), it MUST get its own `prisma/schema.prisma` + `prisma.config.ts` +
> `DATABASE_URL` + migrations directory + local DB. Never point a second product at another product's
> DB, and never add a foreign key across product schemas. Cross-product data crosses an API/contract,
> not a FK (ADR 0038 D1).

**Why:** a shared DB couples products — one product's migration can break another; backups, scaling,
and failure domains entangle; a client product can't be cleanly handed off while its data lives in the
shared DB. BBL's verified lineage graph is the moat and deserves its own failure domain.

### What stays shared (NOT separated)

- **`packages/ui-kit`** — the shared component kernel (m-card, tokens, AdminKanban). ADR 0038 D3.
- **Tooling + CI** — one repo, one toolchain (bun). Separation is about *data, deploys, and brands*,
  not components or tooling.

## Current product DBs

| Product | App dir | Schema | DB (local) | DB (prod) |
| --- | --- | --- | --- | --- |
| **BBL** (flagship) | `apps/web` | `apps/web/prisma/schema.prisma` (~125 models) | `ronindojo_prodsnap` (app, DB-backed Bun, build); `ronindojo_e2e` (Playwright); optional `ronindojo_dev` scratch | Neon (the BBL DB) |
| **Mammoth Build CRM** | `clients/mammoth-build-crm` | `clients/mammoth-build-crm/prisma/schema.prisma` (HubSpot-replacement CRM core) | `mammoth_dev` | *(none yet — Neon at SHIP, Phase 2, operator-gated)* |

### BBL DB formalization

The Postgres that `apps/web` already rides **is** BBL's dedicated database — no data moves, no rename.
ADR 0038's "give BBL its own DB" is satisfied by *declaring* the existing one BBL-owned (the local
mirror is `ronindojo_prodsnap`; prod is the BBL Neon project). Clients get **new** DBs; BBL keeps its.
Do not rename the BBL DB or its env var — that would break every BBL connection for zero benefit.

## Anatomy of a per-app DB

Each product owns this set, scoped to its app dir:

```text
clients/<product>/
  prisma/
    schema.prisma        # own generator + datasource (provider only; URL lives in the config file)
    migrations/          # own migration history, isolated from every other product
  prisma.config.ts       # Prisma 7: the app-owned Migrate URL selection lives HERE
  .env                   # gitignored — the app-owned database URL(s)
  .env.example           # committed template
  .generated/            # gitignored — generated Prisma client (regenerated by `prisma generate`)
```

### Two Prisma-7 gotchas (learned SESSION_0459)

1. **No inline `url` in `schema.prisma`.** Prisma 7 removed `datasource { url = env(...) }`. The
   datasource block carries only `provider`; the Migrate connection URL moves to `prisma.config.ts`.
   A new local-only app may use the minimal `datasource: { url: env("DATABASE_URL") }` form. BBL's
   `apps/web/prisma.config.ts` is intentionally stricter: locally it prefers `DIRECT_URL`, and on Vercel it
   normalizes the direct Neon endpoint, so BBL's `DATABASE_URL` and `DIRECT_URL` must name the same database.
   See Mammoth's
   [`prisma.config.ts`](../../../clients/mammoth-build-crm/prisma.config.ts) (local-first; the Neon
   pooled/direct-URL normalization in `apps/web/prisma.config.ts` is only needed once the app
   provisions Neon, Phase 2).
2. **`prisma.config.ts` disables auto-`.env` loading** — add `import "dotenv/config"` at the top of
   the config file (both `apps/web` and Mammoth do this).

### Package manager: standalone bun per client product

`clients/*` are **deliberately not** in the root `workspaces` (`["apps/*","packages/*"]`). A client
product runs a **standalone `bun install`** in its own dir, producing its own
`clients/<product>/bun.lock` — the root `bun.lock` is untouched. This keeps the dependency failure
domain isolated (a client's bad/vulnerable dep can't break BBL's install or CI) and keeps the product
cleanly extractable on handoff (ADR 0033/0034). The shared `ui-kit` kernel is still consumed in-repo via
a `file:../../packages/ui-kit` dependency. (Decision: SESSION_0459, operator-chosen.)

#### Gotcha: the `file:` ui-kit link breaks Turbopack (whole-dir symlink fix)

A **standalone** `bun install` of the `@ronin-dojo/ui-kit` `file:` dep (whose `package.json` declares
`files: ["src"]`) materializes `node_modules/@ronin-dojo/ui-kit` as a **real directory with an absolute
per-file `package.json` symlink**. Next/Turbopack reads a package via its directory realpath and treats
that per-file symlink as a *redirect*, failing the dev/build with:

```text
Error parsing package.json file .../node_modules/@ronin-dojo/ui-kit/package.json
package.json is not parseable: invalid JSON: a redirect can't be parsed as json
```

`apps/web` never hits this because `workspace:*` gives it a **single whole-directory symlink**, which
Turbopack realpaths cleanly. **Fix (SESSION_0460):** a committed `postinstall` that reshapes the link to
the whole-dir form, idempotently, on every install — see Mammoth's
[`scripts/link-ui-kit.mjs`](../../../clients/mammoth-build-crm/scripts/link-ui-kit.mjs):

```jsonc
// clients/<product>/package.json
"scripts": { "postinstall": "node scripts/link-ui-kit.mjs", ... }
```

`next.config` still needs `transpilePackages: ["@ronin-dojo/ui-kit"]` + `turbopack.root` pinned to the
monorepo top (so the symlink target is inside the resolution root). Any standalone client consuming the
in-repo kernel needs this `postinstall`.

## Adding the next product DB

1. **Scaffold the schema.** Create `clients/<product>/prisma/schema.prisma` with its own `generator`
   (`provider = "prisma-client"`, `engineType = "client"`, `output = "../.generated/prisma"`) and a
   `datasource db { provider = "postgresql" }` (provider only — no `url`). Add only that product's
   models. **No BBL models, no cross-product FKs.**
2. **Add the config file.** Create `clients/<product>/prisma.config.ts` (copy Mammoth's) — it carries
   `datasource: { url: env("DATABASE_URL") }` and `import "dotenv/config"`.
3. **Add deps + scripts.** Add `@prisma/client` (dep) + `prisma` (devDep) + the `db:*` scripts to the
   product's `package.json`. Install standalone: `cd clients/<product> && bun install`.
4. **Add env.** `clients/<product>/.env.example` (committed) + `.env` (gitignored), each with the
   product's own `DATABASE_URL` (a new DB name — never another product's). Gitignore `.env` and
   `/.generated`.
5. **Create the local DB + first migration.**

   ```bash
   /Applications/Postgres.app/Contents/Versions/latest/bin/createdb <product>_dev
   cd clients/<product>
   bunx prisma migrate dev --name init   # creates + applies the migration; auto-handles the shadow DB
   ```

6. **Prove isolation** (the gate before declaring it done) — see below.
7. **Provision prod at SHIP** (operator-gated): create the product's own Neon project, wire
   `DATABASE_URL`/`DIRECT_URL` in its Vercel project, and grow `prisma.config.ts` to mirror
   `apps/web/prisma.config.ts`'s Neon pooled/direct normalization. **Never** run `--accept-data-loss`
   against prod; prod migrates via `prisma migrate deploy` (see
   [schema-migration](schema-migration.md) §"Production migration").

## Isolation proof (do this on every new product DB)

A new product's migration must create tables **only** in its own DB and leave every other product's DB
**byte-identical**. Snapshot before, migrate, diff after:

```bash
PSQL=/Applications/Postgres.app/Contents/Versions/latest/bin/psql
TABLES="SELECT table_name FROM information_schema.tables WHERE table_schema='public' ORDER BY table_name;"

# BEFORE: snapshot the BBL DB (and any other product DB)
$PSQL ronindojo_prodsnap -tAc "$TABLES" > /tmp/bbl_before.txt

# ... run the new product's `prisma migrate dev` ...

# AFTER: re-snapshot and diff — an EMPTY diff is the proof
$PSQL ronindojo_prodsnap -tAc "$TABLES" > /tmp/bbl_after.txt
diff /tmp/bbl_before.txt /tmp/bbl_after.txt && echo "BBL DB UNCHANGED ✅"
$PSQL <product>_dev -tAc "$TABLES"   # shows ONLY the new product's tables + _prisma_migrations
```

**Reference proof (SESSION_0459, Mammoth Phase 1):** `mammoth_dev` got its 10 CRM tables (+
`_prisma_migrations`); `ronindojo_prodsnap` stayed at 140 public tables with an identical sorted-table
digest; `ronindojo_dev` unchanged; root `bun.lock` unchanged. That empty diff is the ADR 0038 Phase-1
deliverable.
